You’ve understood enums correctly—and you’ve just hit an important design constraint 👍

✅ Enums are immutable (by design)
❌ You should not (and practically cannot) change their internal data at runtime

So this part of your idea:

“Developers using my system should be able to change file paths”

👉 does NOT belong inside the enum

🧠 The key design principle

Think of it like this:

Enum = identity (what cursor it is)
Data (like file paths) = configuration (how it looks)

👉 You separate identity from configuration

✅ Correct pattern (used in real engines)
1. Enum = fixed cursor types
public enum CursorType {
    ALIAS,
    COPY,
    POINTER,
    WAIT
}

No file paths here. No mutable data.

2. Store customizable data outside the enum
private final Map<CursorType, CursorInformation> cursors = new EnumMap<>(CursorType.class);
3. Load defaults
private void loadDefaults() {
    cursors.put(CursorType.ALIAS,
        new CursorInformation(new Point(72, 20), "alias.png"));

    cursors.put(CursorType.COPY,
        new CursorInformation(new Point(12, 4), "copy.png"));
}
4. Allow overrides (this is what you want)
public void overrideCursor(CursorType type, String newPath) {
    CursorInformation old = cursors.get(type);

    cursors.put(type,
        new CursorInformation(
            old.getHotspot(),
            newPath,
            old.isAnimated(),
            old.getFrameDelay()
        )
    );
}
5. Usage by game developers
customCursor.overrideCursor(CursorType.ALIAS, "my_modded_alias.png");
customCursor.setCursor(CursorType.ALIAS);
🔥 Why this is the right architecture
✅ Enum stays stable and safe
✅ Users can customize freely
✅ No hacks or mutating constants
✅ Clean separation of concerns
✅ Matches how real systems (game engines/UI frameworks) work
⚠️ What NOT to do
enum CursorType {
    ALIAS("alias.png"); // ❌ don’t expect to change this later
}

Because then you’ll want to mutate it—and that’s where enums fight you.

💡 If you want even more flexibility

You can even allow custom cursors outside the enum:

public void registerCustomCursor(String name, CursorInformation info)

Then support both:

setCursor(CursorType.ALIAS);
setCursor("modded-cursor");
🧠 Final takeaway

You were very close to a solid architecture.

👉 The correct split is:

Enum → “what cursor is this?”
Map → “how is it implemented?”
